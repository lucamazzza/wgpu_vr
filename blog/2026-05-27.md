# 2026-05-27

After some research on the possible architecture this project will study, this is the following decisions/suppositions
I made:

1. The technologies included are: `WebGPU` (through `wgpu_native`), `OpenXR`
2. There will be two prototypes that showcase how those two work as standalone technologies (one for `wgpu` and one for
`openxr`)
3. The integration between the two will be the main part of the project that will include:
    * The creation of a shared swapchain (info down below)
    * The introduction of synchronization on the WebGPU rendering loop
    * The introduction of stereoscopic rendering in WebGPU pipeline
4. From this a final demo, encompassing all of the VR course features will be developed
5. Then analysis of performance and effort for the user + guidelines

## Swapchain

In native C++ there is no way to interoperate WebGPU with OpenXR from either of the technologies' APIs; currently
supported are only: `XR_KHR_opengl_enable`, `XR_KHR_vulkan_enable` and `XR_KHR_D3D11_enable`/`XR_KHD_D3D12_enable`.
No specification for `XR_KHR_webgpu_enable` exists yet.

Given that WebGPU "does not exist" for OpenXR, we have to boil down to the "Least Common Denominator", in this case
either Vulkan, DirectX11/12 or Metal (which has no support whatsoever for OpenXR) to share a swapchain of textures
to pass between WebGPU and OpenXR.

This would consist of a bridge library which does not expose the underlaying use of Vulkan, and allows the passing of
information between the two.

The swapchain would simply extract a `VKImage` from the OpenXR struct and import it in WebGPU.
